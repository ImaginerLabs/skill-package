# PRD 增补提案 — Epic 6：UX 体验全面优化

**Author:** John (PM) / Alex
**Date:** 2026-04-12
**类型:** 现有产品增补 PRD（基于 Chrome MCP 实地体验审计）
**依赖 Epic:** Epic 0 ~ Epic 5（全部完成后可并行启动）

---

## Executive Summary

本提案基于对 Skill Manager v0.1.0 的**全页面实地体验审计**（通过 Chrome DevTools MCP 逐一走查所有功能模块），系统性归纳了 6 类用户体验痛点。

这些痛点不是功能缺失，而是**已有功能之间的"胶水层"缺失**——数据展示不一致、操作引导不连贯、信息密度失衡——导致用户在使用时产生困惑和摩擦。

**核心洞察：** 产品功能已经完整，但用户体验的"最后一公里"尚未打通。Epic 6 的目标是让每一个已有功能都能被用户**顺畅发现、正确理解、高效使用**。

**预计规模：** 4 个 Story，1 名前端开发者，预计 1.5 周完成。

---

## 审计方法

- **工具：** Chrome DevTools MCP（截图 + A11y 树分析）
- **覆盖页面：** Skill 库、工作流、同步、导入、路径配置、设置（全部 6 个页面）
- **审计维度：** 信息一致性、操作引导、视觉层次、空状态设计、跨页面联动

---

## 痛点清单（按优先级排序）

### P0 — 阻断性问题（用户会困惑）

#### 痛点 1：分类计数与实际 Skill 归属严重脱节

**现象：** 侧边栏显示「编程开发 0」「文档写作 0」「DevOps 0」，但主区域明明展示了 3 个 Skill 卡片（代码审查助手、CI/CD 流水线配置、技术文档写作）。

**根因：** 这 3 个 Skill 的 Frontmatter `category` 字段值（`coding`、`devops`、`writing`）与分类配置中的 `id` 字段不匹配，导致分类归属计算为 0，Skill 全部落入「全部」分类。

**用户影响：** 用户点击「编程开发」分类，看到空列表，误以为没有相关 Skill，分类系统形同虚设。这是**最高优先级**的体验问题。

**截图证据：** 侧边栏「全部 3」vs「编程开发 0」「文档写作 0」「DevOps 0」

---

#### 痛点 2：工作流页面缺少「已有工作流」入口

**现象：** 进入 `/workflow` 页面，直接呈现空白编排器（工作流名称输入框 + 空画布），没有任何已创建工作流的列表视图。

**根因：** 工作流列表视图可能存在于其他位置（如 Skill 库的「工作流」分类），但工作流编排页本身没有提供「查看/管理已有工作流」的入口。

**用户影响：** 用户每次进入工作流页面都是「新建」状态，无法快速找到并编辑已有工作流。用户需要返回 Skill 库 → 切换到工作流分类 → 找到工作流 → 点击编辑，路径过长。

---

### P1 — 体验摩擦（用户需要额外思考）

#### 痛点 3：同步页左右分栏信息密度严重失衡

**现象：** 同步页 `/sync` 采用左右分栏布局：左侧「选择 Skill」区域内容充实（分类树 + Skill 列表），右侧「同步目标（0）」区域完全空白，仅有一个「添加第一个同步目标」按钮。

**用户影响：** 新用户进入同步页，视觉重心在左侧，但实际上**必须先配置右侧的同步目标才能使用左侧的选择功能**。操作顺序与视觉引导相反，造成认知负担。用户可能先勾选 Skill，点击「开始同步」，才发现提示「请先选择要同步的 Skill」，但真正的阻塞是「没有同步目标」。

---

#### 痛点 4：导入页路径输入与路径预设功能割裂

**现象：** 导入页 `/import` 只有一个文本输入框（默认值 `~/.codebuddy/skills`）+ 扫描按钮，没有接入「路径配置」页面已有的路径预设功能。

**用户影响：** 用户在「路径配置」页面辛苦配置了路径预设，但在导入页面完全用不上，每次都要手动输入路径。两个功能模块之间存在明显的数据孤岛。

---

### P2 — 体验优化（用户会感谢）

#### 痛点 5：全局搜索结果缺少描述摘要

**现象：** Command Palette（⌘K）搜索结果展示 Skill 名称 + 分类标签（如「代码审查助手 coding」），但没有描述摘要。

**用户影响：** 当多个 Skill 名称相似时（如「代码审查助手」「代码质量检查」），用户无法通过搜索结果直接判断哪个是目标 Skill，必须逐一点击查看预览，效率低下。

---

#### 痛点 6：顶部栏「未配置」状态缺乏分步引导

**现象：** 顶部栏右上角显示「未配置同步目标，点击配置」按钮，点击后跳转到同步页 `/sync`，但同步页没有高亮或引导用户「下一步应该点击右侧的添加目标按钮」。

**用户影响：** 用户被引导到同步页后，面对左右分栏的复杂界面，不知道该先做什么，引导链断裂。

---

## 功能需求（新增 FR）

### 分类系统修复与增强

- **FR-E6-01：** 系统在 Skill 导入/创建时，自动将 Frontmatter `category` 字段值与分类配置 `id` 进行规范化匹配（大小写不敏感、别名映射），确保分类计数准确
- **FR-E6-02：** 侧边栏分类计数实时反映实际归属 Skill 数量，与主区域展示结果完全一致
- **FR-E6-03：** 当 Skill 的 `category` 字段无法匹配任何已知分类时，将其归入「未分类」虚拟分类，并在 UI 中提示用户修复

### 工作流页面改进

- **FR-E6-04：** 工作流页面顶部新增「已有工作流」列表区域（折叠式或标签页），展示所有已创建的工作流 Skill，支持直接点击进入编辑模式
- **FR-E6-05：** 工作流列表提供「新建工作流」按钮，点击后清空编排器进入新建模式，与「编辑已有工作流」模式明确区分

### 同步页引导优化

- **FR-E6-06：** 同步页在「同步目标为空」状态下，右侧面板展示醒目的引导卡片，明确说明「第一步：添加同步目标」→「第二步：选择 Skill」→「第三步：开始同步」的操作顺序
- **FR-E6-07：** 「开始同步」按钮在「无同步目标」状态下，tooltip 提示「请先在右侧添加同步目标」，而非「请先选择要同步的 Skill」（当前提示具有误导性）

### 导入页路径预设集成

- **FR-E6-08：** 导入页扫描路径输入框旁新增「从预设选择」下拉按钮，列出路径配置页面已保存的所有路径预设，点击即可填入输入框
- **FR-E6-09：** 若路径预设为空，下拉菜单展示「前往路径配置添加预设」快捷入口

### 全局搜索增强

- **FR-E6-10：** Command Palette 搜索结果每条记录新增描述摘要（截取 `description` 字段前 60 字符），在 Skill 名称下方以次要文字色展示
- **FR-E6-11：** 搜索结果支持按分类分组展示（Skills 分组 / 工作流分组 / 页面分组），提升结果可读性

### 顶部栏引导优化

- **FR-E6-12：** 顶部栏「未配置」按钮点击后，跳转到同步页并自动触发「添加同步目标」对话框（或高亮右侧添加按钮），完成引导链闭环

---

## 非功能需求

- **NFR-E6-01：** 分类计数计算不得影响 Skill 列表加载性能（< 200ms，500 Skill 规模）
- **NFR-E6-02：** Command Palette 搜索结果渲染时间 < 100ms（含描述摘要）
- **NFR-E6-03：** 所有新增 UI 元素遵循现有暗色主题设计系统（Code Dark + Run Green）

## 用户旅程（改进后）

### 旅程 A：新用户首次配置同步（改进后）

1. 用户看到顶部栏「未配置同步目标」→ 点击
2. **[改进]** 自动跳转到同步页并弹出「添加同步目标」对话框
3. 用户填写目标路径 → 保存
4. 右侧同步目标列表出现第一个目标，**[改进]** 引导卡片消失，左侧 Skill 选择区域高亮提示「现在选择要同步的 Skill」
5. 用户勾选 Skill → 点击「开始同步」→ 成功

### 旅程 B：用户编辑已有工作流（改进后）

1. 用户进入工作流页面
2. **[改进]** 顶部展示已有工作流列表（如「代码提交前检查」「文档生成流程」）
3. 用户点击目标工作流 → 编排器加载该工作流的步骤
4. 用户修改步骤顺序 → 点击「保存工作流」→ 覆盖保存
5. 全程无需离开工作流页面

### 旅程 C：用户通过搜索快速定位 Skill（改进后）

1. 用户按 ⌘K 打开 Command Palette
2. 输入「review」
3. **[改进]** 搜索结果展示：「代码审查助手 — 帮助进行代码审查，检查代码质量... coding」
4. 用户一眼识别目标 Skill → 点击 → 直接跳转到预览

| 分类 ID 规范化可能影响现有 Skill 文件的 category 字段 | 🟡 中 | 只做读取时规范化，不修改源文件；提供迁移提示 |
