# Product Requirements Document — Sidebar 重设计与交互优化

**Author:** Alex
**Date:** 2026-04-13
**关联主 PRD:** `prd.md`（Skill Manager）

---

## Executive Summary

本 PRD 描述对 Skill Manager 左侧边栏（Sidebar）的一次系统性重设计，涉及三个相互关联的改动：

1. **分类导航迁移**：将原 Sidebar 中的「分类」入口从主导航中移除，改为在「Skill 库」页面内以**二级 Sidebar** 的形式展示，使分类成为 Skill 库的子级导航，而非顶层导航项。

2. **Sidebar 重定位为系统状态面板**：原 Sidebar 导航区域空出的位置，改为展示系统级统计信息（Skill 数量、工作流数量等），并引入类似 GitHub 贡献热力图（豆点图）的**活跃度可视化组件**，直观呈现用户的 Skill 修改频率。

3. **分类管理 Tab 动画优化**：「分类管理」页面中的「分类设置 / 套件管理」Tab 切换，增加**滑块平移动画**，提升交互质感。

**核心价值：** 让 Sidebar 从单纯的导航工具进化为「系统状态仪表盘」，同时通过结构调整使分类浏览与 Skill 内容在空间上更紧密结合，降低认知跳转成本。

---

## 背景与问题分析

### 现状

当前 Sidebar 结构（参见 `Sidebar.tsx`）：

- 顶层导航：Skill 库 / 工作流 / 同步 / 导入 / 路径配置 / **分类**
- 分类目录树（`CategoryTree` 组件）嵌入 Sidebar 底部，作为 Skill 库的子级筛选器

### 问题

| #   | 问题                                                        | 影响                               |
| --- | ----------------------------------------------------------- | ---------------------------------- |
| 1   | 「分类」作为顶层导航项，与 Skill 库的分类目录树存在语义重叠 | 用户困惑：两处「分类」入口职责不清 |
| 2   | Sidebar 仅承担导航职责，空间利用率低                        | 用户无法快速感知系统整体状态       |
| 3   | 分类管理 Tab 切换为硬切换，无过渡动画                       | 交互质感粗糙，与整体 UI 风格不符   |

## 需求详情

### 需求 1：分类导航迁移至 Skill 库二级 Sidebar

#### 用户故事

> 作为一名 Skill Manager 用户，当我进入 Skill 库时，我希望在页面左侧看到分类目录，这样我可以在不离开 Skill 库页面的情况下快速切换分类筛选，而不需要通过顶层导航跳转到独立的「分类」页面。

#### 功能需求

- **FR-S1-1**：从主 Sidebar 导航列表中移除「分类」导航项（原 `{ to: "/settings", icon: Settings, label: "分类" }`）。
- **FR-S1-2**：当用户处于 Skill 库页面（路由 `/`）时，在主 Sidebar 右侧展示一个**二级 Sidebar**，内容为现有的 `CategoryTree` 组件。
- **FR-S1-3**：二级 Sidebar 仅在 Skill 库页面激活时可见；切换到其他页面（工作流、同步等）时，二级 Sidebar 自动收起/隐藏。
- **FR-S1-4**：二级 Sidebar 的宽度独立于主 Sidebar，建议固定宽度（参考现有 `CategoryTree` 宽度），与主 Sidebar 之间有明显的视觉分隔线。
- **FR-S1-5**：分类管理（CRUD）入口保留在「设置」或独立的「分类管理」页面，二级 Sidebar 仅作为**只读筛选导航**，不提供分类编辑能力。

#### 交互说明

```
主 Sidebar（固定）          二级 Sidebar（条件显示）
┌─────────────────┐         ┌──────────────────────┐
│ 📚 Skill 库 ●  │ ──────▶ │ 全部                 │
│ ⚡ 工作流       │         │ ├─ coding             │
│ 🔄 同步         │         │ ├─ writing            │
│ 📥 导入         │         │ └─ workflow           │
│ 📁 路径配置     │         │                      │
│ ⚙️ 设置         │         │ [分类管理入口]        │
└─────────────────┘         └──────────────────────┘
```

### 需求 2：Sidebar 重设计为系统状态面板

#### 用户故事

> 作为一名 Skill Manager 用户，我希望在侧边栏中能快速看到我的 Skill 库整体状态（有多少 Skill、多少工作流），以及我最近的活跃情况，就像 GitHub 首页的贡献热力图一样，让我对自己的 Skill 资产有直观感知。

#### 功能需求

**统计信息区块（Stats Panel）**

- **FR-S2-1**：在主 Sidebar 导航列表下方（分隔线之后），展示系统统计信息区块，包含：
  - Skill 总数（排除工作流类型）
  - 工作流总数（`type: workflow` 的 Skill 数量）
  - 分类总数
- **FR-S2-2**：统计数字应实时反映当前数据状态，在用户执行导入、删除等操作后自动更新。
- **FR-S2-3**：统计区块采用紧凑布局，每项统计包含图标、数字和标签，不占用过多垂直空间。

**活跃度热力图（Activity Heatmap）**

- **FR-S2-4**：在统计信息区块下方，展示一个**活跃度豆点图**（类 GitHub Contribution Graph），用于可视化用户的 Skill 修改频率。
- **FR-S2-5**：热力图的数据来源为 Skill 文件的**最后修改时间**（`fs.stat().mtime`），统计维度为**过去 12 周**（约 3 个月）的每日活跃度。
- **FR-S2-6**：热力图的颜色深浅表示当日修改的 Skill 文件数量：
  - 0 次：最浅色（背景色）
  - 1-2 次：浅绿
  - 3-5 次：中绿
  - 6+ 次：深绿（主题色 `--primary`）
- **FR-S2-7**：鼠标悬停在某个豆点上时，显示 Tooltip，内容为「YYYY-MM-DD · N 次修改」。
- **FR-S2-8**：热力图宽度自适应 Sidebar 宽度，豆点大小和间距根据可用宽度动态计算（优先保证 12 周完整显示）。
- **FR-S2-9**：热力图数据在应用启动时计算一次，并在用户执行文件操作（导入、同步、删除）后触发重新计算。

#### 视觉规范

│  ⚡ 工作流                  │

### 需求 3：分类管理 Tab 滑块平移动画

#### 用户故事

> 作为一名 Skill Manager 用户，当我在「分类管理」页面切换「分类设置」和「套件管理」两个 Tab 时，我希望看到一个平滑的滑块动画，而不是生硬的内容切换，这样的交互体验更流畅、更现代。

#### 功能需求

- **FR-S3-1**：「分类管理」页面的 Tab 切换组件（「分类设置」/「套件管理」），在 Tab 激活状态切换时，底部/背景滑块应以**平移动画**过渡，而非瞬间跳变。
- **FR-S3-2**：动画时长建议 `200ms`，缓动函数使用 `ease-in-out`，与项目现有 `transition-colors duration-200` 风格保持一致。
- **FR-S3-3**：滑块动画使用 CSS `transform: translateX()` 实现，避免使用 `left/right` 属性动画（性能更优）。
- **FR-S3-4**：Tab 内容区域的切换（分类设置内容 ↔ 套件管理内容）可使用淡入淡出（`opacity` 过渡）或直接切换，不强制要求内容区域动画。
- **FR-S3-5**：动画在低性能设备上应能优雅降级（遵循 `prefers-reduced-motion` 媒体查询，若用户开启了减少动画偏好，则跳过动画直接切换）。

#### 参考视觉（当前状态 → 目标状态）

```
当前（硬切换）：
[分类设置] [套件管理]   →   [分类设置] [套件管理]
 ████████                              ████████

目标（滑块平移）：
[分类设置] [套件管理]   →   [分类设置] [套件管理]
 ████████                    ────────▶  ████████
                              （200ms 平移动画）
```

## 非功能需求

- **NFR-S1**：二级 Sidebar 的显示/隐藏切换应有平滑过渡动画（`transition` + `opacity/transform`），避免布局抖动。
- **NFR-S2**：热力图渲染不应阻塞主线程，数据计算在后端 API 完成，前端仅负责渲染。
- **NFR-S3**：热力图组件在 Sidebar 宽度变化时（如窗口缩放）应响应式重新布局，不出现溢出或截断。
- **NFR-S4**：所有新增动画遵循 `prefers-reduced-motion` 媒体查询，保障无障碍访问。
- **NFR-S5**：本次改动不影响现有路由结构和其他页面的功能。
