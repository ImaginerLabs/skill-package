## 成功与亮点

### 1. 全栈功能完整交付

Epic 3 实现了完整的工作流编排功能闭环：从 Skill 选择 → 拖拽排序 → 步骤编辑 → 文件生成预览 → 保存/更新/删除。前后端协同良好，API 设计清晰。

**具体成果：**

- 前端 6 个组件（`WorkflowEditor`、`SkillSelector`、`StepList`、`StepItem`、`WorkflowPreview`、`WorkflowList`）职责划分清晰
- 后端 `workflowService.ts` 实现了完整的 CRUD + 预览功能
- `workflowRoutes.ts` 遵循了路由层薄封装原则，所有请求体通过 `WorkflowSchema` Zod 校验

### 2. 架构一致性优秀

Epic 3 的实现严格遵循了项目既定的架构规范：

- ✅ 服务端函数式导出（无 class）
- ✅ 路由层 Zod safeParse 校验
- ✅ `safeWrite()` 原子写入
- ✅ `isSubPath()` 路径安全校验（deleteWorkflow 中）
- ✅ 统一 `ApiResponse<T>` 响应格式
- ✅ 前端通过 `src/lib/api.ts` 封装 API 调用
- ✅ Zustand store 状态管理（`workflow-store.ts`）
- ✅ 共享类型定义在 `shared/types.ts`
- ✅ 服务端 import 带 `.js` 扩展名

### 3. 拖拽排序实现优雅

`@dnd-kit` 集成干净利落：

- `StepList.tsx` 使用 `DndContext` + `SortableContext` 包裹
- `StepItem.tsx` 使用 `useSortable` hook
- 同时支持鼠标拖拽和键盘排序（Alt+↑/↓）
- 拖拽激活距离 5px 防止误触
- 拖拽中视觉反馈（opacity + shadow + border 高亮）

### 4. 编辑模式设计巧妙

`workflow-store.ts` 通过 `editingWorkflowId` 字段优雅地区分新建/编辑模式：

- `loadWorkflow()` 加载已有工作流到编排器
- `WorkflowPreview.tsx` 根据 `isEditing` 自动切换"生成工作流"/"更新工作流"按钮
- `reset()` 统一清空所有状态（包括编辑 ID）

### 5. 复用已有基础设施

Epic 3 充分复用了前序 Epic 的基础设施：

- `useSkillSearch` hook（fuse.js 模糊搜索）→ SkillSelector 搜索
- `skill-store.ts` → Skill 列表获取

- `AlertDialog` UI 组件 → 删除确认

## 挑战与改进空间

### 1. Story 3-3 和 3-4 缺少独立 Story 文件

**问题：** Story 3-3（文件生成与预览）和 3-4（工作流管理）没有创建独立的 story 文件，直接进行了实现。这违反了 Story 生命周期规范（backlog → ready-for-dev → in-progress → qa → review → done）。

**影响：**

- 缺少 Dev Agent Record，无法追溯实现决策
- 缺少 QA 阶段记录和 Code Review 记录
- 后续回顾缺少详细的 story 级别数据

**行动项：** 后续 Epic 严格执行 Story 文件创建流程，每个 Story 必须有独立文件。

### 2. WorkflowList 中的 Markdown 解析逻辑

**问题：** `WorkflowList.tsx` 的 `handleEdit` 函数中，使用正则表达式从 Markdown 内容解析工作流步骤：

```typescript
const stepRegex =
  /## Step (\d+)\s*\n\n\*\*使用 Skill:\*\* `([^`]+)`(?:\n\n([\s\S]*?))?(?=\n## Step|\s*$)/g;
```

这段逻辑存在以下风险：

- 正则表达式脆弱，对格式变化敏感（如多余空行、缩进变化）
- 解析逻辑放在前端组件中，违反了"路由层薄封装"的精神（虽然这是前端组件）
- 通过 `skillName` 反查 `skillId` 的逻辑不够健壮（名称可能重复或变更）

**建议：** 后续考虑在后端提供 `GET /api/workflows/:id` 端点，直接返回结构化的工作流数据（包含 steps 数组），而非让前端解析 Markdown。

### 3. 缺少 WorkflowPreview 组件测试

**问题：** 6 个工作流组件中，`WorkflowPreview.tsx` 没有对应的测试文件。该组件包含 API 调用（预览和保存）、loading 状态管理、编辑/新建模式切换等逻辑，是测试覆盖的盲区。

**行动项：** 补充 `WorkflowPreview.test.tsx` 测试。

### 4. 工作流 ID 稳定性

**问题：** 工作流 ID 由文件名 `slugify()` 生成。当用户编辑工作流名称时，如果名称变化导致 slug 变化，但文件名不变，可能出现 ID 与名称不一致的情况。`updateWorkflow` 使用旧 ID 查找文件，这是正确的，但如果未来需要重命名文件，需要额外处理。

**当前状态：** 可接受。更新操作不改变文件名，只改变文件内容。

### 5. 删除操作缺少撤销功能

**问题：** Epic 3 的 AC 中提到"删除操作附带 5 秒撤销按钮（UX-DR14）"，但当前实现使用的是 `AlertDialog` 确认对话框，没有实现撤销功能。

**影响：** 功能偏差，但确认对话框提供了足够的误操作保护。

**行动项：** 如果用户反馈需要撤销功能，可在后续迭代中添加。

## 模式与洞察

### 正面模式

1. **组件职责划分清晰** — 每个组件 100-200 行，单一职责，易于理解和测试
2. **Store 设计前瞻** — `workflow-store.ts` 在 Epic 0 就已完整定义，Epic 3 实现时无需修改 store 接口
3. **UI 组件复用率高** — 大量复用 `ui/` 基础组件和已有的 `shared/` 组件
4. **后端服务遵循既定模式** — `workflowService.ts` 的代码风格与 `skillService.ts`、`importService.ts` 保持一致

### 需关注的模式

1. **Story 文件缺失** — 3-3 和 3-4 跳过了 story 文件创建，需要在后续 Epic 中严格执行
2. **前端解析后端数据** — WorkflowList 中的 Markdown 解析应该是后端的职责

### Epic 4: IDE 同步与路径配置

**目标：** 用户可以配置同步目标，一键同步 Skill 到 IDE 项目目录。

**计划 Story：**

| Story | 名称                   | 依赖        |
| ----- | ---------------------- | ----------- |
| 4-1   | 同步目标路径配置       | Epic 1 基础 |
| 4-2   | Skill 选择与批量操作   | 4-1         |
| 4-3   | 一键同步执行与结果日志 | 4-1, 4-2    |
| 4-4   | 同步状态指示器         | 4-3         |

### 对 Epic 3 的依赖

Epic 4 与 Epic 3 **无直接依赖**。Epic 4 依赖的是 Epic 1 的 Skill 浏览基础设施（skill-store、API 客户端、UI 组件）。

### 准备工作

**技术准备：**

- [ ] 确认 `config/settings.yaml` 中 sync targets 的数据结构
- [ ] 确认 `configService.ts` 是否需要扩展以支持 sync target CRUD
- [ ] 确认文件复制策略（扁平化复制，不保留分类子目录）

**知识准备：**

- [ ] 复习 `safeWrite()` 和 `atomicWrite()` 的使用模式（同步操作需要大量文件写入）
- [ ] 了解 `sync-store.ts` 已有的接口定义

**架构确认：**

- 同步 API 端点已在架构文档中定义：`POST /api/sync/push`
- `SyncTarget`、`SyncResult`、`SyncDetail` 类型已在 `shared/types.ts` 中定义

### 潜在风险

1. **文件复制性能** — NFR3 要求 100 个文件同步 < 2s，需要并行复制
2. **路径安全** — 同步目标是用户配置的绝对路径，需要严格的路径校验
3. **目标目录不存在** — 需要处理自动创建目录的逻辑
4. **同名文件覆盖** — 需要在日志中清晰标注覆盖操作
