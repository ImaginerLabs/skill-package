# 评估驱动开发指南

本文档介绍如何通过"先构建评估，再写文档"的方法论，创建高质量、可验证的 Claude Skill。

---

## 核心理念：先评估，后迭代

传统开发流程：写文档 → 测试 → 发现问题 → 修改

**评估驱动开发流程**：

```
定义评估标准 → 编写 SKILL.md → 运行评估 → 分析差距 → 迭代改进 → 重复
```

优势：基于真实使用数据而非假设来迭代，避免"感觉写得不错但实际效果差"的问题。

---

## 评估结构

每个评估用例包含以下字段：

```json
{
  "skills": ["skill-name"],
  "query": "用户的触发语句",
  "files": ["path/to/test-file.csv"],
  "expected_behavior": "期望 Claude 执行的具体行为描述"
}
```

### 字段说明

| 字段                 | 说明                                           |
| -------------------- | ---------------------------------------------- |
| `skills`             | 期望被触发的 Skill 名称列表                    |
| `query`              | 模拟用户输入的触发语句                         |
| `files`              | 测试所需的附件文件路径（可选）                 |
| `expected_behavior`  | 具体、可验证的期望行为描述（避免模糊表述）     |

### 评估用例示例

```json
[
  {
    "skills": ["data-analyzer"],
    "query": "分析这个 CSV 文件的销售趋势",
    "files": ["test-data/sales_2024.csv"],
    "expected_behavior": "调用 scripts/analyze_data.py，输出包含月度趋势的统计摘要，并生成折线图"
  },
  {
    "skills": ["data-analyzer"],
    "query": "解释什么是数据分析",
    "files": [],
    "expected_behavior": "不触发此 Skill（概念解释，非数据分析任务）"
  },
  {
    "skills": ["pdf-editor"],
    "query": "把这个 PDF 旋转 90 度",
    "files": ["test-data/sample.pdf"],
    "expected_behavior": "调用 scripts/rotate_pdf.py，参数 angle=90，输出旋转后的 PDF 文件"
  }
]
```

## Claude A / Claude B 协作迭代模式

使用两个独立的 Claude 实例进行协作：

```
Claude A（创建者）                    Claude B（测试者）
─────────────────                    ─────────────────
编写 SKILL.md          →  提供给  →  模拟真实用户使用
分析测试结果           ←  反馈  ←   报告实际行为与期望的差距
更新 SKILL.md          →  再次  →  重新测试
```

**Claude A 的职责**：
- 根据需求编写 SKILL.md
- 分析 Claude B 的反馈
- 识别文档中的歧义或缺失信息
- 迭代改进文档

**Claude B 的职责**：
- 不看 Claude A 的设计意图，只看 SKILL.md
- 模拟真实用户的各种触发语句
- 报告实际执行行为
- 标记与 `expected_behavior` 不符的地方

## 评估维度

### 1. 触发准确性

测试 Skill 是否在正确的场景被触发，不在无关场景误触发：

```markdown
应该触发的（正例）：
✅ "分析这个 CSV 文件"
✅ "给我这些数据的统计摘要"
✅ "可视化销售趋势"

不应触发的（负例）：
❌ "写一个数据分析的教程"（创作内容，非分析数据）
❌ "解释什么是数据分析"（概念解释，非执行分析）
❌ "帮我写一首关于数据的诗"（完全无关）
```

### 2. 资源使用率

验证 Claude 是否正确使用了 Skill 提供的脚本和参考文档：

- 是否调用了 `scripts/` 中的脚本？
- 是否读取了相关的 `references/` 文档？
- 是否使用了 `assets/` 中的模板？

### 3. 输出质量

对比实际输出与 `expected_behavior` 的差距：

- 输出格式是否符合预期？
- 关键信息是否完整？
- 是否有多余或错误的内容？

## 迭代改进流程

### 常见问题与修复方向

| 问题现象                     | 可能原因                   | 修复方向                           |
| ---------------------------- | -------------------------- | ---------------------------------- |
| Skill 未被触发               | description 关键词不足     | 补充触发关键词和场景描述           |
| 在无关场景误触发             | description 过于宽泛       | 缩小范围，添加排除条件             |
| 脚本未被使用                 | SKILL.md 中说明不够清晰    | 增加具体的脚本调用示例             |
| 输出格式不符合预期           | 工作流程描述不够具体       | 添加输出格式示例和约束             |
| Claude 自行发明了不存在的方法 | 缺少明确的边界说明         | 在"注意事项"中明确禁止的行为       |

### 迭代最佳时机

- ✅ 使用 Skill 后立即进行（记忆最新鲜）
- ✅ 发现明显问题时
- ✅ 用户需求变化时
- ✅ 定期审查（如每月一次）
