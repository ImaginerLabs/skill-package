## Skill 诊断概述

### 何时需要增强 Skill？

当出现以下情况时，需要对 Skill 进行诊断和优化：

- **触发不准确**：该触发时没触发，不该触发时却触发了
- **执行效果不佳**：输出质量不符合预期
- **覆盖范围不足**：遗漏了重要的使用场景
- **可维护性差**：代码结构混乱，难以扩展
- **用户反馈问题**：实际使用中发现的问题

### 三级加载机制（诊断视角）

| 级别                | 诊断重点                | 检查方法                                      |
| ------------------- | ---------------------- | -------------------------------------------- |
| **Level 1: 元数据** | 触发准确性             | description 是否清晰、是否覆盖关键场景        |
| **Level 2: 指令**   | 执行效果与效率         | 工作流程是否合理、指引是否清晰                |
| **Level 3: 资源**   | 资源完整性与可维护性   | 脚本是否正确、文档是否完整                    |

## 增强流程

```
诊断分析 → 识别问题 → 制定方案 → 执行优化 → 效果验证 → 迭代改进
```

### 第 1 步：诊断与分析

**读取并分析目标 Skill**：

1. 阅读 SKILL.md 完整内容
2. 检查目录结构是否完整
3. 审查 YAML frontmatter 的 name 和 description
4. 识别现有资源文件

**诊断检查清单**：

```
□ SKILL.md 行数是否超过 500 行限制
□ description 是否使用第三人称
□ description 是否包含触发条件说明
□ 工作流程是否使用清单式结构
□ 是否存在冗余或未使用的资源
□ 文档风格是否统一（祈使句/不定式）
```

### 第 2 步：识别改进点

根据诊断结果，确定需要优化的方向：

| 问题类型       | 典型症状                     | 优化方向               |
| -------------- | ---------------------------- | ---------------------- |
| **触发问题**   | 描述模糊、关键词遗漏         | 优化 description       |
| **结构问题**   | 内容过长、嵌套过深           | 拆分文件、渐进式披露   |
| **逻辑问题**   | 步骤混乱、边界不清           | 重构工作流程           |
| **资源问题**   | 脚本缺失、文档不完整         | 补充或优化资源文件     |

### 第 3 步：制定优化方案

**优先级排序**（从高到低）：

1. **P0 - 触发优化**：description 不准确会导致 Skill 无法正常使用
2. **P1 - 结构优化**：内容过长影响上下文效率和可维护性
3. **P2 - 细节完善**：补充示例、优化措辞

**优化方案模板**：

```markdown
## 优化方案

### 问题 1：[问题描述]
- **现状**：[当前实现]
- **目标**：[期望效果]
- **改动**：[具体修改内容]

### 问题 2：[问题描述]
- ...
```

### 第 4 步：执行优化

按照优化方案逐步修改：

1. **备份原文件**：在修改前创建快照
2. **分步修改**：每次只改一个优化点
3. **即时验证**：修改后检查格式和内容
4. **记录改动**：文档化每次修改的原因

### 第 5 步：效果验证

**自检清单**：

```
□ 语法检查：SKILL.md 是否为有效 Markdown
□ 结构检查：YAML frontmatter 是否完整
□ 内容检查：所有引用路径是否正确
□ 风格检查：是否符合写作规范
```

### 第 6 步：迭代改进

根据验证结果，决定是否需要进一步优化：

- ✅ **通过**：优化完成
- ⚠️ **部分通过**：修复剩余问题后重新验证
- ❌ **未通过**：重新分析问题，调整优化方案

## 诊断与评估

### 触发准确性评估

**评估方法**：构建触发测试集

```json
[
  { "query": "用户可能输入的描述", "should_trigger": true/false }
]
```

**评估标准**：

| 指标     | 目标值          | 说明                        |
| -------- | --------------- | --------------------------- |
| 召回率   | ≥ 90%          | 应当触发的场景中实际触发比例 |
| 精确率   | ≥ 85%          | 触发的场景中正确触发的比例   |
| 误触发率 | ≤ 10%          | 不应触发却触发的比例         |

### 执行效果评估

**评估维度**：

1. **正确性**：输出是否满足任务需求
2. **完整性**：是否覆盖了所有必要步骤
3. **效率**：执行路径是否最优
4. **可读性**：输出格式是否清晰

### 结构健康度评估

| 指标         | 优秀 | 良好 | 需改进 |
| ------------ | ---- | ---- | ------ |
| SKILL.md 行数 | <300 | <500 | >500   |
| 嵌套层级     | 1级  | 2级  | >2级   |
| 资源文件数量 | 3-5个| 5-10个| >10个  |

## 优化策略

### Description 优化策略

**核心原则**：清晰、准确、具有区分度

**优化步骤**：

1. **提取核心功能**：用一句话描述 Skill 做什么
2. **列举触发场景**：用户可能如何描述这个需求
3. **添加区分关键词**：区分于相关但不同的 Skill

**优化示例**：

```yaml
# ❌ 优化前
description: "帮助用户写代码的工具"

# ✅ 优化后
description: >
  Assists with code writing, debugging, and refactoring across multiple
  programming languages. This skill should be used when the user mentions
  writing code, fixing bugs, improving code structure, or asking for
  programming help.
```

### 内容结构优化策略

**渐进式披露原则**：

```
SKILL.md（核心流程）→ references/（详细文档）→ scripts/（执行脚本）
```

**拆分标准**：

| 内容类型           | 位置         | 行数建议     |
| ------------------ | ------------ | ------------ |
| 核心工作流程       | SKILL.md     | ≤ 500 行     |
| 详细规格文档       | references/  | < 5k 行      |
| 可执行脚本         | scripts/     | 无限制       |
| 输出模板           | assets/      | 无限制       |

### 工作流优化策略

**清单式工作流**：

```markdown
## 工作流程

- [ ] 第 1 步：[具体步骤]
- [ ] 第 2 步：[具体步骤]
- [ ] 第 3 步：[具体步骤]
```

**反馈循环设计**：

```
执行 → 验证 → 发现问题 → 修复 → 重新验证 → 直至通过
```

### 迭代改进策略

**评估驱动开发**：

1. **构建测试用例**：3-5 个代表性的使用场景
2. **运行效果对比**：优化前 vs 优化后
3. **收集用户反馈**：实际使用中的问题
4. **持续迭代**：根据反馈持续改进

**迭代周期**：

```
评估 → 发现问题 → 制定方案 → 执行 → 验证 → 下一轮评估
```

**迭代终止条件**：

- 用户表示满意
- 所有反馈为空（效果良好）
- 无法取得实质性进展

### 通用指导原则

**避免意外原则（Lack of Surprise）**：

Skill 的内容不应在意图上让用户感到意外。不要创建误导性的 Skill 或旨在促进未授权访问、数据泄露或其他恶意活动的 Skill。

**写作风格建议**：

- 优先使用祈使句形式提供指令
- 避免使用生硬的 MUST 或 NEVER 大写形式
- 尝试解释为什么某事很重要
- 尽量使 Skill 具有通用性，不要过于针对特定示例

### Description 优化指导

Description 字段是决定 Claude 是否调用 Skill 的主要机制。优化 description 可以提高触发准确性。

**优化步骤**：

1. **生成测试查询**：创建 20 个测试查询（触发/不触发混合）
2. **用户审核**：让用户审核测试集
3. **运行优化循环**：根据测试结果迭代优化
4. **应用结果**：更新 SKILL.md frontmatter

**测试查询要求**：

- **应触发查询（8-10个）**：不同措辞覆盖同一意图，包含不明确提及 Skill 名称的场景
- **不应触发查询（8-10个）**：近似的邻近领域、模糊措辞，应触发其他工具的场景

**优化示例**：

```yaml
# ❌ 优化前
description: "帮助用户写代码的工具"

# ✅ 优化后
description: >
  Assists with code writing, debugging, refactoring, and explanation across
  multiple programming languages. This skill should be used when the user
  mentions writing code, fixing bugs, improving code structure, explaining
  code logic, or asking for programming help. Make sure to use this skill
  whenever the user mentions coding tasks, even if they don't explicitly
  ask for a 'coding helper'.
```
