# 暂存区代码测试用例覆盖分析

## 核心能力

在提交前读取暂存区变更，分析本次修改的业务意图与能力变化，针对变更代码从**主流程链路**和**边缘情况**两个维度设计测试用例，并输出可执行的测试分析报告。

## 适用场景

- 功能迭代后验证测试覆盖是否充分
- 重构代码后确认行为一致性
- Bug 修复后防止回归
- 新功能上线前的测试设计评审

---

## 执行流程

### Step 1：读取暂存区 Diff

```bash
git diff --staged
```

获取所有已暂存文件的变更内容，作为后续分析的输入。

### Step 2：梳理本次修改目的与能力变化

精读 diff 内容，向用户清晰说明：

**2.1 修改目的**：从函数命名、注释、变量名、业务逻辑推断核心意图，识别是新增功能、Bug 修复、重构优化还是配置调整。

**2.2 能力变化对比**：

| 维度     | 改动前                | 改动后              |
| -------- | --------------------- | ------------------- |
| 新增能力 | —                     | 描述新增的功能/行为 |
| 删除能力 | 描述被移除的功能/行为 | —                   |
| 修改行为 | 原有行为描述          | 新行为描述          |
| 影响范围 | 列出受影响的模块/接口 | —                   |

### Step 3：识别技术栈与测试框架

根据变更文件特征自动识别技术栈：

| 文件特征                            | 技术栈          | 推荐测试框架                 |
| ----------------------------------- | --------------- | ---------------------------- |
| `*.tsx` / `*.jsx` + React import    | React           | Jest + React Testing Library |
| `*.ts` / `*.js`（纯逻辑）           | TypeScript / JS | Jest / Vitest                |
| `*.vue`                             | Vue             | Vitest + Vue Test Utils      |
| `*.go`                              | Go              | `testing` 标准库 + testify   |
| `*.py`                              | Python          | pytest                       |
| `*.java` / `*.kt`                   | Java / Kotlin   | JUnit 5 + Mockito            |
| `*.spec.*` / `*.test.*`（已有测试） | 任意            | 沿用现有框架，补充缺失用例   |

### Step 4：设计测试用例

**4.1 主流程链路测试（Happy Path）**

覆盖核心功能的正常执行路径：

- 输入合法数据时，函数/组件的预期输出或行为
- 关键业务流程的端到端验证
- 状态变更的正确性
- 返回值/副作用的准确性

设计原则：每个新增/修改的公共函数至少一条主流程用例，覆盖最常见的业务场景（80% 的使用频率），验证核心输出，不过度断言实现细节。

**4.2 边缘情况测试（Edge Cases）**

| 边缘类型         | 典型场景                                         |
| ---------------- | ------------------------------------------------ |
| **空值/缺省**    | `null`、`undefined`、空字符串、空数组、空对象    |
| **边界值**       | 最大值、最小值、零值、负数、超长字符串           |
| **异常输入**     | 类型错误、格式非法、超出范围的枚举值             |
| **并发/竞态**    | 重复调用、快速连续触发、异步操作未完成时再次触发 |
| **网络/IO 异常** | 接口超时、返回错误码、网络中断、部分数据缺失     |
| **权限/状态**    | 未登录、无权限、资源不存在、状态机非法跳转       |
| **副作用**       | 函数调用后是否产生预期外的状态变更或副作用       |

**4.3 回归测试（针对 Bug 修复）**

若本次变更为 Bug 修复，额外设计：

- 复现原始 Bug 的测试用例（确认修复有效）
- 验证修复不影响相关功能的回归用例

### Step 5：代码层面分析与验证

**5.1 可测性分析**：函数是否为纯函数（易测）或含副作用（需 Mock），是否依赖外部状态/接口，是否存在难以测试的耦合点。

**5.2 Mock 策略**：针对外部依赖给出具体 Mock 方案，参考 `references/test-examples.md` 中的 Mock 示例。

**5.3 断言要点**：

- ✅ 断言**行为结果**（返回值、状态变更、DOM 变化）
- ✅ 断言**副作用**（函数调用次数、调用参数）
- ❌ 避免断言**内部实现细节**（私有变量、中间状态）

**5.4 静态代码分析验证测试用例**

读取变更文件的**完整源码**（不仅是 diff，而是整个函数/模块的实现），逐条对照 Step 4 设计的测试用例，通过代码逻辑推导验证每条用例的预期结果是否与实现一致，给出验证结论：

| 结论标记 | 含义                                                      |
| -------- | --------------------------------------------------------- |
| ✅ 通过  | 代码逻辑与预期一致，该用例在正确实现下应当通过            |
| ❌ 失败  | 代码逻辑与预期矛盾，该用例会失败，说明存在 Bug 或设计偏差 |
| ⚠️ 存疑  | 代码逻辑存在歧义、依赖运行时状态或外部 Mock，无法静态确认 |

验证步骤：

1. 使用 `read_file` 读取 diff 中涉及的完整源文件（不只是变更行）
2. 逐条取出 Step 4 中的测试用例，模拟其输入，在脑中执行代码逻辑
3. 对比执行结果与用例的预期输出，得出 ✅ / ❌ / ⚠️ 结论
4. 对 ❌ 用例：说明代码哪一行/哪段逻辑导致失败，并给出修复建议
5. 对 ⚠️ 用例：说明不确定的原因（如依赖接口返回值、环境变量等）

### Step 6：输出测试报告

按照 `references/output-template.md` 中的格式输出完整的测试分析报告。

## 注意事项

- **读取完整源码**：Step 5.4 验证阶段需读取完整文件内容（非仅 diff），以确保逻辑推导准确
- **不修改源代码**：本 Skill 仅做分析与建议，不对暂存区代码进行任何修改
- **聚焦变更范围**：只针对 diff 中实际变更的代码设计用例，不对未变更代码做全量分析
- **优先级标注**：P0 为阻断性用例（必须通过），P1 为重要用例，P2 为补充用例
- **可执行性**：所有测试用例示例代码均基于实际 diff 内容生成，确保可直接参考实现
