# 代码综合审查工作流

执行四阶段审查，产出结构化 Markdown 报告。每个阶段独立完成，前置阶段的阻断性问题不阻塞后续阶段。

## 阶段概览

| 阶段 | 名称 | 核心目标 | 输入 | 输出 |
| ---- | ---- | -------- | ---- | ---- |
| 1 | 基础代码审查 | 发现阻断性问题（安全、错误处理、边界） | 代码/diff | 问题清单（阻断/建议） |
| 2 | 代码质量检查 | 深度分析（Lint、健壮性、性能） | 代码 + Stage 1 结果 | 质量问题清单 |
| 3 | 体积优化分析 | 评估 Bundle Size 影响 | 代码 + 依赖声明 | 优化建议清单 |
| 4 | 测试覆盖与验证 | 设计用例 + 静态验证 | 代码 + diff | 测试用例 + 验证结论 |

## 审查目标识别

按优先级判断审查模式：

1. `--commit=<hash>` → 读取该 commit 的 diff
2. `--staged` → 读取暂存区变更
3. `--path=<path>` 或 `--file=<file>` → 读取指定文件/目录的完整代码
4. `--focus=<area>` → 可选，仅执行指定维度（quality / bundle / test）
5. 无参数 → 询问用户选择模式

### 代码获取方式

| 模式      | 获取方式                                              | 审查重点      |
| --------- | ----------------------------------------------------- | ------------- |
| Commit    | `git show <hash> --stat` + `git show <hash>`          | diff 变更部分 |
| 暂存区    | `git diff --staged --name-only` + `git diff --staged` | diff 变更部分 |
| 文件/目录 | `read_file` / `list_dir` 后读取                       | 完整代码内容  |

Commit/暂存区模式还需用 `git show <hash>:<file>` 读取关键文件的完整上下文，避免 diff 孤岛效应。

## Stage 1: 基础代码审查

**目标**：发现调试遗留、敏感信息、语法问题、边界遗漏。

执行步骤：

1. 读取目标代码，梳理变更文件清单
2. 识别技术栈（React/Vue/Go/Python 等）
3. 按 `references/checklist.md` 中的「基础检查项」逐项扫描
4. 执行技术栈专项检查（见 checklist.md §2）
5. 问题按严重程度分为 **阻断性** 和 **建议项**

关键原则：

- 敏感信息泄露（硬编码密钥/Token）必须标为阻断性
- 调试代码遗留（console.log/debugger）标为建议项，除非在关键路径上

---

## Stage 2: 代码质量检查

**目标**：深度分析健壮性、副作用、Lint 问题、性能隐患。

执行步骤：

1. 运行 Lint 检查（使用 `read_lints` 工具，或项目自带的 lint 命令），重点关注：
   - `no-undef`、`no-unused-vars`、`exhaustive-deps`（高危）
   - `no-explicit-any`、`rules-of-hooks`（中危）
2. 按 `references/checklist.md` §3 评估健壮性、副作用、性能
3. Commit/暂存区模式仅分析变更部分；文件模式分析完整代码

关键原则：

- 竞态条件和内存泄漏（未清理的订阅/定时器）是高频问题，务必检查
- 修改公用组件时，需要用 `search_content` 检查下游调用方是否受影响

---

## Stage 3: 体积优化分析

**目标**：评估变更对 Bundle Size 的影响，给出分级优化建议。

执行步骤：

1. 从 diff 或 import 语句识别新增依赖
2. 按 `references/checklist.md` §4 检查依赖引入方式、冗余代码、懒加载机会
3. 用分级标准（🔴 >10KB / 🟡 2~10KB / 🟢 <2KB gzip）给出建议

关键原则：

- 全量引入大型库（lodash/moment/antd 全量）是常见问题，优先标记
- 体积分析只在变更引入新依赖或大型代码块时才有意义，纯逻辑修改可简述

---

## Stage 4: 测试覆盖与验证

**目标**：设计主流程和边缘情况的测试用例，并通过静态代码分析验证测试与实现的一致性。

执行步骤：

1. 梳理本次修改的目的和能力变化
2. 识别技术栈对应的测试框架（见 checklist.md §5）
3. 设计主流程链路测试（Happy Path）和边缘情况测试
4. 提供 Mock 策略建议
5. **静态代码分析验证测试用例**（关键步骤）

关键原则：

- 测试用例设计应覆盖修改涉及的能力变化，而非面面俱到
- 优先 P0（主流程）和 P1（关键边缘），P2 仅列举
- **必须对设计的测试用例进行代码验证**，不能只设计不验证

### 4.5 静态代码分析验证测试用例

**目的**：读取变更文件的完整源码，逐条对照设计的测试用例，通过代码逻辑推导验证每条用例的预期结果是否与实现一致。

验证结论标记：

| 结论标记 | 含义                                                      |
| -------- | --------------------------------------------------------- |
| ✅ 通过  | 代码逻辑与预期一致，该用例在正确实现下应当通过            |
| ❌ 失败  | 代码逻辑与预期矛盾，该用例会失败，说明存在 Bug 或设计偏差 |
| ⚠️ 存疑  | 代码逻辑存在歧义、依赖运行时状态或外部 Mock，无法静态确认 |

验证步骤：

1. 使用 `read_file` 读取 diff 中涉及的完整源文件（不只是变更行）
2. 逐条取出设计的测试用例，模拟其输入，在脑中执行代码逻辑
3. 对比执行结果与用例的预期输出，得出 ✅ / ❌ / ⚠️ 结论
4. 对 ❌ 用例：说明代码哪一行/哪段逻辑导致失败，并给出修复建议
5. 对 ⚠️ 用例：说明不确定的原因（如依赖接口返回值、环境变量等）

**特别注意**：

- 必须读取完整源码进行验证，而非仅凭 diff 推断
- ❌ 结论是重要发现，应在报告的"必须处理"部分重点标注
- ⚠️ 结论需标注为"需运行时验证"，不能忽略

## 报告输出

使用 `references/report-template.md` 中的模板生成报告。核心设计原则：

1. **行动项前置** — 报告最前面就是「必须处理」和「建议优化」，让用户第一时间知道该做什么
2. **阻断性问题带代码 diff** — 不只说问题，直接给出修复建议的代码对比
3. **详情在后** — 四阶段的详细分析结果放在行动项之后，供需要深入了解时查阅
4. **评分精简** — 每个维度一句话说明扣分原因，不做冗长的逐项打分

### 评分规则

| 维度         | 权重 | 评分依据                                                    |
| ------------ | ---- | ----------------------------------------------------------- |
| 基础代码审查 | 25%  | 无阻断性问题满分，每个阻断 -10，每个建议 -3                 |
| 代码质量检查 | 30%  | Lint 错误、健壮性、性能问题综合评估                         |
| 体积优化分析 | 20%  | 按高/中/低优先级建议数量扣分                                 |
| 测试用例覆盖 | 25%  | 全部 ✅ 通过满分，有 ❌ 失败用例 -20，有 ⚠️ 存疑 -10        |

评分分级：🟢 优秀 90-100 / 🟡 良好 70-89 / 🔴 需改进 <70

**测试覆盖评分细则**：
- 全部 ✅ 通过：100 分
- 有 ⚠️ 需运行时验证的用例（无法静态确认）：-10 分
- 有 ❌ 代码逻辑与预期不符的用例（代码逻辑错误）：-20 分

## 执行策略

### 工具使用建议

| 工具 | 用途 | 使用场景 |
| ---- | ---- | -------- |
| `read_file` | 读取代码文件 | 批量读取，避免逐文件串行 |
| `read_lints` | 获取 IDE 诊断 | Lint 检查，比手动分析更准确 |
| `search_content` | 搜索引用 | 评估修改的下游影响 |
| `git show` | 读取历史版本 | Commit 模式下补充完整上下文 |

### 执行顺序建议

**推荐顺序**：Stage 1 → Stage 2 → Stage 3 → Stage 4

**原因**：
1. Stage 1 识别阻断性问题，需要优先发现
2. Stage 2 依赖 Stage 1 的基础审查结果
3. Stage 3 和 Stage 4 可并行执行（各自独立分析）

**并行优化**：当变更涉及多个不相关文件时，可以并行执行 Stage 2-4 的分析。

### 常见问题模式识别

快速识别以下高频问题模式：

| 模式 | 特征 | 严重性 |
| ---- | ---- | ------ |
| 内存泄漏 | useEffect 无清理、订阅未取消、定时器未清除 | 阻断 |
| 敏感信息泄露 | 硬编码 token/secret/password | 阻断 |
| 错误被吞 | catch 块空实现、只 console.error 不处理 | 阻断 |
| 竞态条件 | 异步操作无 AbortController、fast-click 重复提交 | 阻断 |
| 类型安全 | `any` 类型滥用、缺少类型守卫 | 建议 |
| 性能问题 | 大列表无虚拟化、同步 heavy 操作 | 建议 |

### 大规模变更的处理

当变更涉及超过 10 个文件时：

1. 先用 `git show --stat` 或 `git diff --staged --stat` 获取全局视图
2. 优先审查核心业务逻辑文件（组件、hooks、工具函数）
3. 配置文件、类型定义、样式文件可简要说明
4. 报告中标注「因变更规模较大，部分文件为简要审查」
